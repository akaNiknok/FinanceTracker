## Decisions
- 2026-06-25: **Budgets schema redesigned → targets-only sheet; actuals computed in code.** Owner disliked the old Budgets sheet. Problems removed: it stored the same target 3× (`Target %`/`USD`/`PHP`), pinned its **own** stale `PHP to USD` rate (59 vs live ~61.3 — a second FX source that drifts), kept sheet-computed actuals/remaining/%/sparkline (the service layer already derives actuals from the ledger), and carried an `Essentials & Rewards` roll-up **row** that `api_getBudgets` would double-count. New `Budgets` = plan only, one row/segment: `Segment | Period | Target Type | Target | Currency | Notes`. New `Budgets.gs` `api_getBudgets` resolves **hybrid** targets (owner choice): `Percent` rows of `MONTHLY_INCOME_PHP` (×3 if Quarterly) for Essentials/Rewards/Stability; `Amount` USD cap for Growth, converted at the **live** FX (owner choice — one FX source, no pinned rate). Actuals = Σ `Expense` `Amount (PHP)` by Segment over the period window (current month, or **calendar quarter** for `Period=Quarterly` — fixes Growth's quarterly $200 investing cap that the old monthly grid mangled). Returns per-segment target/actual/remaining/pctUsed/isOver + an `essentialsRewards` roll-up (the segments owner actually tracks; Stability unfunded — no dedicated savings acct yet). Old `Monthly Expenses` table split into its own **`Recurring`** sheet (`Description, Currency, Amount, Transaction Fee, Months Left, Group`; govt block→`Group=Govt`) via `api_getRecurring` — pure notes, no calc. New Script Property **`MONTHLY_INCOME_PHP`** (default 47200) = planning base for percent targets (`cfgMonthlyIncomePhp_`). Migration `Migration.setupBudgets()`: extracts Monthly Expenses→Recurring, backs up + rebuilds Budgets with 4 seed rows, sets the income property; idempotent (skips if `Target Type` header present / `Recurring` exists). Sheet-only change — no redeploy. Verify with `Tests.gs test_budgets`. Dashboard + bootstrap now use the computed budgets. **Owner still to run** `setupBudgets` in the GAS editor after `npm run push`.
- 2026-06-25: **Accounts `Type` derived from `Subtype`, not from balance sign.** Owner asked whether to infer Asset/Liability from sign — rejected: their convention stores liabilities **positive** (RCBC +44,169 owed) so "negative=liability" misclassifies; sign is transient, account nature is stable; Type also gates interest/credit-limit logic. Better fix (removes the redundant manual column): `Migration.setupAccountType()` creates an `AccountType` reference tab (`Subtype→Type`: Credit→Liability, else Asset) + sets Accounts `Type` to a **per-row** VLOOKUP (per-row not ARRAYFORMULA because Accounts is a Google Sheets Table; Tables reject whole-column array formulas but auto-fill per-row on new rows). Idempotent, backs up Accounts, logs blank-Subtype accounts + unseen Subtypes. Service layer unchanged (reads Type). Schema-hygiene guidance also recorded: treat `Name`/`Category` as **immutable FKs** (renames orphan ~950 tx) — biggest latent rework risk; keep deriving (never duplicate). Also added `Migration.setupDataValidation()` — dropdowns for Accounts.Subtype (←AccountType), Accounts.Interest Frequency (fixed list), Transactions.Category (←Categories); lenient by default (`MIG_VALIDATION_STRICT=false`: flags, doesn't block — service layer stays the hard validator; flip to true to reject manual typos).
- 2026-06-24: **Balances now read from the sheet, not recomputed in code.** Owner's Accounts tab has `Current Balance` (native) + `Current Balance (PHP)` formula columns (= Starting Balance + Σ tx, live FX for USD, Google Finance for Shares) — these are the real derivation, so `api_getAccounts` just reads them (fields: `balancePhp` liabilities-positive, `balanceNative` =qty for Shares, `netWorthPhp` signed, `availableCredit`, `isLiability`, `isShares`). This supersedes the earlier "computed for PHP / stored for FX-Shares" approach and fixes Wise/IBKR/Shares (my JS recompute summed historical `Amount (PHP)` and was wrong off-PHP; the sheet sums native + converts). Consistent with the ARRAYFORMULA philosophy: derivation in the Sheet, service reads inputs/derived. `Tests.gs test_balanceReconciliation` repurposed to a **native-currency** integrity check (Starting + Σ native deltas vs sheet `Current Balance`) — flags only genuine ledger↔sheet disagreements. **Result (owner ran it): ALL 26 accounts reconcile diff=0** incl. USD (Wise 197.61) and Shares (VWRA 4.009) — ledger fully agrees with the sheet's balance formulas. The probe inverts deltas for **liability** accounts (credit cards track amount-OWED positive; a charge increases it) so RCBC reconciles too. Net worth uses `netWorthPhp` (liabilities negative). Receivables are **assets, positive = that person owes the user** (Berry +34,424). Starting Balance is **native currency** (Wise start=20 USD).
